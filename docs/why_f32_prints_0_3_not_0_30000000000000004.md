# Kenapa `f32` Bisa Tampil `0.3` (Bukan `0.30000000000000004`) di Zig 0.16

Kasus yang kamu lihat:

```text
0.1 + 0.2 = 0.3
one_plus_two : 0.30000000000000000000
one_plus_two as f32 0.30000000000000000000
one_plus_two as f64 0.30000001192092896000
c = 0.30000001192092896000
d = 0.30000001192092896000
c bits = 0x3e99999a
d bits = 0x3e99999a
0.30000000000000004
```

## Jawaban Singkat

Ini **bukan bug**.  
`0.1 + 0.2` dalam `f32` memang disimpan sebagai nilai biner terdekat (`0x3e99999a`), yang nilai desimal tepatnya sekitar `0.30000001192092896`.  
Tapi formatter default Zig untuk `{}` / `{d}` memilih representasi desimal paling pendek yang tetap *round-trip* ke nilai float yang sama, dan untuk bit pattern ini string terpendeknya adalah **`"0.3"`**.

## Kenapa `as f64` Menjadi `0.30000001192092896000`?

Karena kamu melakukan:

- hitung dulu di `f32`
- lalu cast hasil `f32` ke `f64`

Cast ini **tidak menghitung ulang** `0.1 + 0.2` dalam presisi `f64`; hanya memperlebar representasi nilai `f32` yang sudah dibulatkan sebelumnya.

Jadi angka error `f32` jadi terlihat jelas saat dicetak lebih presisi di `f64`.

## Kenapa Bisa Muncul `0.30000000000000004`?

Itu biasanya muncul saat operasi dilakukan langsung di `f64`, misalnya:

```zig
const x = @as(f64, 0.1) + @as(f64, 0.2);
std.debug.print("{d:.17}\n", .{x}); // 0.30000000000000004
```

`f64` punya error representasi yang berbeda dari `f32`, sehingga bentuk desimal panjangnya juga berbeda.

### Elaborasi Engineering: Kenapa beda ini penting?

#### A) Dari sisi System Engineering

- **Precision budget berbeda**  
  `f32` punya 24 bit precision signifikan, `f64` punya 53 bit.  
  Artinya dua sistem yang "logikanya sama" bisa menghasilkan jejak angka desimal yang beda kalau satu pakai `f32`, yang lain `f64`.

- **Observability/logging bisa menipu**  
  Kalau log default pakai shortest format, nilai `f32` bisa terlihat "rapi" (`0.3`) padahal internalnya `0x3e99999a`.  
  Untuk debugging numerik, log juga bit pattern atau log dengan precision tetap.

- **Reproducibility test**  
  Snapshot test berbasis string float raw sering rapuh lintas compiler/runtime/arsitektur.  
  Lebih aman:
  - bandingkan dengan epsilon (`abs(a-b) < eps`)
  - atau bandingkan bit pattern jika memang butuh exact binary equality.

- **Boundary behavior**  
  Keputusan bisnis berbasis threshold (contoh `value >= 0.3`) bisa beda jika satu pipeline menghitung di `f32` lalu cast, pipeline lain langsung `f64`.

#### B) Dari sisi Protocol Engineering

- **Wire format float vs decimal string**  
  Jika protokol kirim float IEEE754 biner (`f32`/`f64`), nilai yang diterima deterministik per bit.  
  Jika kirim string desimal (`"0.3"`), endpoint penerima bisa parse ke tipe berbeda (`f32` vs `f64`) dan menghasilkan bit berbeda.

- **Canonicalization penting**  
  Untuk sistem yang butuh signature/hash (mis. event signing, consensus log), representasi angka harus dikunci:
  - tipe wajib (`f32` atau `f64`)
  - aturan serialisasi (mis. shortest-roundtrip atau fixed precision)
  - aturan parse yang konsisten.

- **Cross-language interoperability**  
  Bahasa A mungkin default ke `f64`, B mungkin menyimpan sebagai `f32` di model tertentu.  
  Tanpa spesifikasi protokol yang eksplisit, angka sama secara "human readable" bisa tidak sama secara bit-level.

- **Backward compatibility**  
  Mengubah field protocol dari `f32` ke `f64` bukan sekadar "lebih presisi"; bisa mengubah output string, hash payload, deduplikasi, dan behavior query/filter.

#### C) Praktik aman yang biasanya dipakai

- Nyatakan tipe numerik eksplisit di schema/protocol (`f32` atau `f64`, jangan implisit).
- Tentukan format serialisasi angka (binary IEEE754 atau decimal canonical).
- Untuk keuangan/kuantitas diskrit, pertimbangkan fixed-point/integer (mis. cents sebagai `i64`) agar tidak kena noise floating point.

## Bukti Bit Pattern Kamu Konsisten

Kamu dapat:

- `c bits = 0x3e99999a`
- `d bits = 0x3e99999a`

Artinya `c` dan `d` benar-benar nilai `f32` yang sama.  
Dan parsing `"0.3"` ke `f32` juga menghasilkan bit pattern yang sama (`0x3e99999a`), jadi formatter Zig sah menampilkan `0.3` sebagai bentuk terpendek yang tetap tepat saat di-parse kembali ke `f32`.

## Rule of Thumb

- `{}` / `{d}`: tampilan ringkas (shortest round-trip), cocok untuk output umum.
- `{d:.N}`: paksa jumlah digit desimal tertentu, cocok untuk analisis numerik.
- Untuk debug nilai float internal, cek juga bit pattern:

```zig
std.debug.print("bits=0x{x}\n", .{@as(u32, @bitCast(value_f32))});
```
